Current State

The prototype is retired as the product architecture.

The current custom hyperedit.js implementation taught the right lessons, but the next version should not continue expanding it. Hyperspace should become a lightweight layer over HyperclayJS: ordinary HTML review artifacts, opt-in editable regions, basic annotation tools, and a local server that persists the saved page.

What carries forward

HTML as source

The saved artifact is a single HTML file in the user's project. It can contain content, state, interaction markup, and annotations.

Human edits

Users should be able to open the page, edit clearly marked regions, and leave comments or visual notes directly on the rendered document.

Local persistence

A trusted localhost server is the correct boundary for file writes, path checks, backups, and atomic saves.

What changes

Retired

Whole-root editing

The base page is not editable by default. Editing must be opt-in at the element level.

Retired

Custom save stack

Hyperspace should not maintain a separate save payload, snapshot, dirty-check, or edit-mode system when HyperclayJS already provides those primitives.

New

Hyperclay runtime

HyperclayJS is the browser runtime. Hyperclay Local's server core is the preferred local host, ideally reused without the Electron desktop shell.

New

Tool layer

Hyperspace defines toolbar actions, annotation commands, and AI-facing DOM tools that compose Hyperclay primitives.

Minimal implementation target

  1. A project-local HTML file can be served by Hyperclay Local's server core, with no custom save stack in Hyperspace.
  2. Only elements marked with editable attributes can receive direct text edits.
  3. Users can add, focus, move, edit, and delete basic comments through direct comment interactions and keyboard deletion.
  4. Basic comments are inserted into the closest document container based on the user's first click; they are not attached to a specific target object.
  5. Runtime overlay UI is excluded from saved HTML.
  6. The host accepts Hyperclay's save request and writes the page back to the correct file.

Annotation model candidate

A comment attribute on arbitrary elements is tempting, but it mixes user feedback into source content and does not scale to multiple notes, resolved state, visual callouts, or range highlights. The cleaner baseline is a nearby HTML comment element inserted into the closest comment host. Its placement in the source is semantic; its placement on screen is CSS. 

<section data-hs-comment-host>
  <button editmode:contenteditable>Confirm</button>

  <aside data-hs-comment movable style="transform: translate(72px, 120px)">
    <p editmode:contenteditable>Label may need to be more specific.</p>
  </aside>
</section>

Current code status

The existing hyperedit.js remains useful reference code, but it is no longer the target architecture. server.js currently follows the Hyperclay-compatible local save contract as a development shim. The current hyperspace.js and hyperspace.css layer provides targeted text editing through small edit indicators, double-click text edits, a design-doc inspired comment/save toolbar, and durable nearby comments with click-to-focus, hold-to-move, double-click edit, keyboard delete, Excalifont plain red text-only rendering, automatic comment sizing, and quiet focus-loss autosave. The preferred product direction is still to reuse Hyperclay Local's server core without the Electron app.
