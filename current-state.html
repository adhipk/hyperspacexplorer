Current State

What hyperedit.js does today.

hyperedit.js is the current browser include that turns one configured page root into a lightweight editing surface. It injects its own toolbar, uses native contenteditable for text edits, wraps selected text for highlights and comments, tracks a small undo stack, and saves either to localStorage or a configured backend URL.

Runtime shape

Current include

The file is an immediately invoked function expression. It avoids double mounting when window.HyperEdit already exists. This describes the current include, not a required source layout.

One editable root

data-root chooses the editable root. Without it, the script falls back to body.

Shadow DOM toolbar

Runtime controls live in a Shadow DOM host appended to document.body, separate from the page markup.

Source boundary

The project can move to modular source files for tools, state, rendering, persistence, and serialization, but the near-term cleanup should not introduce a bundler or generated build step. The useful minimum is typed plain JavaScript, clear local boundaries, and fewer hidden cross-dependencies.

Configuration

<script
  src="/hyperedit.js"
  data-root="main"
  data-autostart="true"
  data-save-url="/__hyperedit/save"
  data-storage-key="custom-key"></script>

Tool behavior

Edit

Toggling edit mode makes the configured root editable, remembers the previous contenteditable and spellcheck values, and restores them when editing stops.

Select

Select is the neutral tool. Double-clicking an editable area returns to select mode and focuses the editable root.

Highlight

Mouse selection plus the highlight tool wraps selected DOM in a mark with a HyperEdit id, color name, pale background, and inherited text color. Shift while highlighting also opens a comment box.

Comment

Comment wraps selected DOM in a span anchor, creates an absolutely positioned editable comment box inside the root, and mirrors box text into the anchor's data-hx-comment attribute.

Annotations

The panel lists highlights and comment anchors, shows snippets, can focus a target, create or reopen its comment box, delete one annotation, or remove all annotations from the root.

Color sample

Sampling replaces the currently selected palette color. Native EyeDropper is preferred; the fallback samples computed background color, then text color.

Undo

Undo restores prior root.innerHTML snapshots. The stack is capped at 40 entries, and a post-save undo can return to the last saved root HTML.

Save

Dirty edits schedule a local draft after 450ms. Manual save serializes the whole document, emits hyperedit:before-save, then either POSTs the payload or stores a cleaned draft locally.

Known constraints

Monolithic implementation

The current source file mixes UI rendering, state, event routing, editing, annotations, persistence, serialization, and public API. That is a current-state observation, not a product requirement. The first improvement should be modularity with type safety, not a build pipeline.

Type safety is missing

The script is plain JavaScript without JSDoc data contracts or static checking. Adding those contracts would reduce risk before larger source-file splits.

Single-root editing

The script only treats the configured root as editable. Current docs use data-root="main", so headers are outside the edit surface. data-hx-ignore is present in markup but is not currently interpreted by the script.

Comment boxes live in content

Comment boxes are appended inside the editable root. Save strips their editable attributes, but the boxes themselves remain part of serialized page HTML.

Full document serialization

Save clones document.documentElement. HyperEdit UI is removed, but unrelated runtime DOM from other scripts or extensions is not categorically excluded.

Range handling is browser-DOM based

Highlights and comments use Range.extractContents() and insertNode(). Complex selections across nested structures are not yet normalized into a stable semantic model.

Reference server is local-only

The Bun server validates paths and extensions, but it does not add authentication, authorization, CSRF protection, audit logs, or merge conflict handling.

Tested behavior

  1. Documentation pages load with the injected toolbar.
  2. Edit mode toggles contenteditable on main.
  3. The edit toggle does not shift the heading layout.
  4. Serialization removes temporary editor state.
  5. Saving without a backend stores a cleaned local draft.
  6. The save hook POSTs serialized HTML to a configured URL.
  7. Comments create inline floating comment boxes.
  8. Existing highlights can receive comments.
  9. The annotations panel can list, delete, and clear annotations.
  10. Floating comment boxes can be moved.
  11. Undo can restore the previous document after save.

Current read

The project is a working prototype for a drop-in HTML editor. The interaction loop is proven enough to continue. The immediate plan is typed source cleanup first; saved source format, comment overlay placement, and tool/event boundaries remain unresolved architecture issues for the following passes.
