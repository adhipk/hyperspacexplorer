Current State

The prototype is retired as the product architecture.

The current custom hyperedit.js implementation taught the right lessons, but the next version should not continue expanding it. Hyperspace should become a lightweight layer over HyperclayJS: ordinary HTML files, opt-in editable regions, annotations, tool definitions, and a local server that persists the saved page.

What carries forward

HTML as source

The saved artifact is a single HTML file in the user's project. It can contain content, state, interaction markup, and annotations.

Human edits

Users should be able to open the page, edit clearly marked regions, and leave comments or visual notes directly on the rendered document.

Local persistence

A trusted localhost server is the correct boundary for file writes, path checks, backups, and atomic saves.

What changes

Retired

Whole-root editing

The base page is not editable by default. Editing must be opt-in at the element level.

Retired

Custom save stack

Hyperspace should not maintain a separate save payload, snapshot, dirty-check, or edit-mode system when HyperclayJS already provides those primitives.

New

Hyperclay runtime

The server injects a pinned HyperclayJS config URL, probably starting from a standard or smooth-sailing preset.

New

Tool layer

Hyperspace defines toolbar actions, annotation commands, and AI-facing DOM tools that compose Hyperclay primitives.

Minimal implementation target

  1. A project-local HTML file can be served without adding script boilerplate to the source.
  2. Only elements marked with editable attributes can receive direct text edits.
  3. Any element can receive an annotation through the toolbar.
  4. Basic comments are inserted into the closest document container based on the user's first click; they are not attached to a specific target object.
  5. Runtime overlay UI is excluded from saved HTML.
  6. The server accepts Hyperclay's save request and writes the page back to the correct file.

Annotation model candidate

A comment attribute on arbitrary elements is tempting, but it mixes user feedback into source content and does not scale to multiple notes, resolved state, visual callouts, or range highlights. The cleaner baseline is a nearby HTML comment element inserted into the closest comment host. Its placement in the source is semantic; its placement on screen is CSS. 

<section data-hs-comment-host>
  <button editmode:contenteditable>Confirm</button>

  <aside data-hs-comment style="--hs-x: 72px; --hs-y: 120px">
    <p>Label may need to be more specific.</p>
  </aside>
</section>

Current code status

The existing hyperedit.js remains useful reference code, but it is no longer the target architecture. server.js now follows the Hyperclay-compatible local save contract for the new Hyperspace slice.
