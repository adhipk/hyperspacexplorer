Phase 2

Save edited HTML through the serving backend.

Phase 2 should keep the client simple. The page edits itself with native browser behavior, then sends a cleaned HTML snapshot to the server that originally served the file.

Client contract

The script dispatches hyperedit:before-save before persistence, then POSTs a JSON payload when data-save-url is configured.

Payload

{
  "pathname": "/index.html",
  "title": "HyperEdit Project Notes",
  "html": "<!DOCTYPE html>...",
  "source": "hyperedit",
  "updatedAt": "2026-06-11T06:00:00.000Z"
}

Backend responsibilities

  1. Resolve the request pathname to an allowed file path.
  2. Reject writes outside the project document root.
  3. Persist the submitted HTML atomically.
  4. Return a clear success or failure response for the toolbar.

Hyperclay-style shape

The backend can behave like a Hyperclay server: serve static HTML, accept edited HTML for the current route, validate the target, write the file, and immediately serve the updated document on refresh.
