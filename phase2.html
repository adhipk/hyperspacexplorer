Phase 2

Save edited HTML through a generic save endpoint.

Phase 2 proves that HyperEdit can keep a simple browser include while delegating persistence to a trusted writer. The source can still be organized without adding a build step; the Bun server is only one minimal backend implementation.

Current contract

The script dispatches hyperedit:before-save, then POSTs a JSON payload when data-save-url is configured. The server validates the target path and writes the submitted HTML.

Prototype payload

{
  "pathname": "/index.html",
  "title": "HyperEdit Project Notes",
  "html": "<!DOCTYPE html>...",
  "source": "hyperedit",
  "updatedAt": "2026-06-11T06:00:00.000Z"
}

Backend responsibilities

  1. Serve ordinary project files from the document root.
  2. Resolve the save target to an allowed file path.
  3. Reject writes outside the project document root.
  4. Persist only accepted HyperEdit save payloads.
  5. Return a clear success or failure response for the toolbar.

What we learned

  1. Native contenteditable is useful for the base editing surface.
  2. Tool UI and comment editors should stay outside the editable document flow.
  3. Persisting raw browser DOM can capture extension code and runtime UI that HyperEdit does not own.
  4. Tool behavior needs explicit input and output contracts before adding more features.

Later save shape

After the typed source cleanup, a later iteration should save HyperEdit-owned artifacts rather than arbitrary browser HTML: tool events, annotation records, or semantic components that can be rendered back onto a clean document.

Future tool boundary

Each tool should declare the browser events it consumes, the state it reads, and the artifact it emits. Shared pointer, selection, keyboard, and save dispatchers should route work to tools instead of each tool wiring global handlers itself.
