Phase 2

Save edited HTML through a generic save endpoint.

Phase 2 keeps the client simple. The page edits itself with native browser behavior, then sends a cleaned HTML snapshot to any configured save endpoint.

Client contract

The script dispatches hyperedit:before-save before persistence, then POSTs a JSON payload when data-save-url is configured. The Bun server in this repo is one minimal implementation of that generic contract.

Payload

{
  "pathname": "/index.html",
  "title": "HyperEdit Project Notes",
  "html": "<!DOCTYPE html>...",
  "source": "hyperedit",
  "updatedAt": "2026-06-11T06:00:00.000Z"
}

Backend responsibilities

  1. Resolve the request pathname to an allowed file path.
  2. Reject writes outside the project document root.
  3. Persist the submitted HTML atomically.
  4. Return a clear success or failure response for the toolbar.

Known issues

  1. Refreshing after a save can shift layout because persisted comment boxes are ordinary document nodes until the editor script restyles them.
  2. Comment boxes sit inside the contenteditable root, so keyboard input can split boxes, create duplicate boxes, or overwrite text.
  3. Full-document serialization can capture browser extension DOM or other third-party injected nodes.
  4. The current script is a working prototype, but its tool handling and event handling need to be separated before more features are added.

Next shape

The next iteration should save HyperEdit-owned artifacts rather than raw browser HTML: tool events, annotation records, or semantic components that can be rendered back onto the document without preserving unrelated runtime DOM.
