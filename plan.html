Reset Plan

Build on Hyperclay, not around it.

The prototype proved the workflow: AI can generate ordinary HTML, users can edit and annotate it in place, and the edited page can be saved back to disk. The product target is agent-authored HTML artifacts for review, not a general purpose website editor. HyperclayJS owns editing, snapshots, dirty checks, persistence attributes, and save behavior, while Hyperspace provides the toolbar, annotation model, AI-facing tool definitions, and small glue needed to run on top of Hyperclay Local's server core.

Standing decisions

Accepted

Opt-in editing

The base HTML page is not contenteditable. Only elements marked as editable through Hyperclay-compatible attributes can be edited by the user.

Accepted

Hyperclay core

HyperclayJS owns the editing and saving stack: edit mode, snapshots, dirty detection, DOM persistence helpers, save state, and the /_/save request shape.

Accepted

Plain source HTML

A source HTML file sits in a user's project. It may be generated by AI, but it should remain understandable and usable without a generated build pipeline.

Accepted

Reuse local server

Prefer Hyperclay Local's server core without the Electron app. The project-local server.js is a compatibility shim for testing, development, and proving runtime injection.

Package boundary

Layer Responsibility
Source HTML Content, interactive elements, editable regions, durable state, and tool markup authored by a user or AI.
HyperclayJS Editing primitives, persistence attributes, utility functions, snapshots, and save calls.
Hyperspace client Basic toolbar UI, annotation interactions, style layer, and tool definitions that compose Hyperclay primitives.
Server core Hyperclay Local's server behavior: local file serving, owner cookies, /_/save, path validation, backups, and writes. Hyperspace should reuse or wrap this core rather than maintain a competing implementation.

User loop

  1. An AI agent or user creates a normal HTML file in a project.
  2. The file marks only intended user-editable regions as editable.
  3. The user opens the page through Hyperclay Local's server core or a compatible development shim.
  4. The host provides HyperclayJS and the Hyperspace toolbar, either through source includes or response-time injection.
  5. The user edits marked regions and adds annotations to any element.
  6. Hyperclay captures the save snapshot and posts it to /_/save.
  7. The server validates the target, writes a backup, and persists the updated HTML file.

Annotation direction

Keep annotations as real HTML close to the content they describe. A basic comment is not attached to a particular object; the first click chooses the closest reasonable document container, and the comment element is inserted there. Hyperclay's movable attribute and a durable transform own its rendered position, so the comment can float anywhere while the source stays near the document area it belongs to. 

<section data-hs-comment-host>
  <h2 editmode:contenteditable>Editable heading</h2>

  <aside data-hs-comment movable style="transform: translate(72px, 120px)">
    <p editmode:contenteditable>Ask the user whether this framing is right.</p>
  </aside>
</section>

For text-range highlights, wrapping the selected text is acceptable because the range itself needs a durable DOM anchor. A basic comment does not need that anchor. Hyperclay helpers such as el.nearest can help tools find the closest data-hs-comment-host while keeping source readable.

Latest milestone

  1. The runtime now injects a design-doc inspired icon toolbar with a comment tool and save button only. Idle selection is implicit, and editable regions advertise themselves with small Lucide pencil indicators over the component.
  2. Explicit editable regions can be edited through the pencil indicator or by double-clicking the text in place, without enabling whole-page edit mode.
  3. Basic comments can be created, focused with a single click, moved by holding and dragging, edited with a double click, and deleted from the keyboard while selected. They render as plain red Excalifont text only, without a card, border, background, text shadow, or manual resize box.
  4. Saved HTML strips runtime controls, transient edit state, and fixed comment width/height while preserving durable comment HTML.
  5. Focus loss now queues a quiet autosave. Manual saves can still show visible feedback, but autosaves should not trigger popups.

Next slice

  1. Dogfood the review loop on agent-created artifacts: generate a page, add comments, edit marked regions, save, reload, and have the agent read the resulting HTML.
  2. Test the Hyperspace client against Hyperclay Local's server core without the Electron app, and keep the current server.js only as a lightweight compatibility shim if needed.
  3. Package the browser runtime and local host path without changing the source HTML convention or adding a generated build step.

Open questions

Annotation anchor format

Prefer existing id values when available. Generate data-hs-id only when the target has no stable identifier.

Text-range persistence

Decide whether range annotations use inline wrappers only, or a wrapper plus a selector record that can be reattached after an agent rewrites the surrounding element.

Tool definition shape

Each tool needs a small contract for DOM inputs, Hyperclay primitives used, persistent output, and runtime-only UI.
