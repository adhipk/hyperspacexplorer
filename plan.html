Decision status

Accepted

Single script frontend

HyperEdit remains a drop-in JavaScript file that injects its UI into ordinary HTML pages.

Accepted

Generic save endpoint

The frontend posts to a configurable data-save-url. The Bun server is only a reference implementation.

Rejected

Raw DOM as source of truth

The live browser DOM can include extension nodes and runtime UI, so it should not be the long-term save format.

Next

Owned artifacts

Save structured HyperEdit artifacts, then reconcile them with clean source HTML.

Goal

Build a single JavaScript file that turns ordinary HTML pages into a lightweight editing surface with minimal page-author setup.

  • Inject the toolbar and runtime UI.
  • Use native contenteditable for text edits.
  • Keep the design close to the minimalist highlighter sketch.
  • Persist through a trusted writer when one exists.

Current result

Working loop

Edit mode, highlights, comments, color sampling, undo, and save are usable enough to validate the interaction direction.

Server answer

A browser can keep a local draft, but source-file writes need a server, app shell, extension permission, or another trusted writer.

Reference server

The Bun server serves root files and implements one minimal /__hyperedit/save endpoint.

Frontend contract

<script
  src="/hyperedit.js"
  data-root="main"
  data-save-url="/your-save-endpoint"></script>

Known issues

Maintainability

hyperedit.js mixes tool behavior, global input events, state updates, serialization, and UI rendering.

Comment editing

Comment boxes currently live inside the editable root. Keyboard input can split boxes, duplicate nodes, overwrite nearby text, and shift layout after refresh.

Serialization risk

Full-document serialization can persist extension DOM, runtime UI, and other nodes HyperEdit does not own.

Asset sizing

SVGs opened as standalone documents need safe intrinsic sizing because page CSS does not apply there.

Tool model

Select

Consumes pointer and selection state. Emits active target state.

Text Edit

Consumes native contenteditable input. Emits content operations for the editable block.

Highlight

Consumes a normalized text range and selected color. Emits an annotation record or semantic marker.

Comment

Consumes a range or annotation id plus isolated overlay input. Emits a comment record.

Color Sample

Consumes EyeDropper output or sampled styles. Emits selected color state.

Save

Consumes dirty HyperEdit-owned state. Emits a structured save request, not arbitrary browser HTML.

Save format comparison

Option Strength Risk Status
Operation log Auditable and rejects unsafe changes server-side. Needs stable anchors and replay logic. Candidate
Annotation records Fits highlights and comments without overlay DOM. Anchors must survive edits. Preferred
Semantic elements Readable source with clear HyperEdit ownership. Needs careful contenteditable behavior. Preferred
Full HTML snapshot Simple for the current prototype. Persists arbitrary browser DOM. Prototype only

Shared event layer

  1. One pointer router dispatches events to the active tool.
  2. One selection reader normalizes browser ranges.
  3. One keyboard/input controller separates page editing from overlay editing.
  4. One save dispatcher builds payloads from HyperEdit-owned state.

Recommended next phase

Move toward a hybrid of annotation records and semantic inline markers. Keep comment editors and tool UI outside the editable root, save structured HyperEdit artifacts, and let the server reconcile those artifacts with clean source HTML.
