Progress and next work.

The active path is typed plain JavaScript with small local boundaries. HyperEdit still ships as a direct browser include; source modularity can grow only where it removes real friction.

Current status

The browser include, editor behaviors, save payload shape, and local draft flow were kept compatible with the existing pages.

Bun owns the workflow

Completed milestones

  1. Added the current-state artifact so the existing JavaScript, server role, save behavior, and known limits are documented outside the source file.
  2. Adopted Bun for package management and project scripts, including the generated lockfile.
  3. Added JSDoc data contracts for options, annotations, state, colors, save payloads, and save results.
  4. Enabled static checking for plain JavaScript without introducing TypeScript output, a bundler, or generated runtime files.
  5. Isolated typed helper boundaries for selectors, annotation lookup, annotation panel rendering/actions, color parsing, serialization cleanup, and save/draft/event flow.
  6. Added typed history and input-event boundaries for undo capture, root snapshot restore, saved-root fallback, and editable input target checks.
  7. Wired project HTML artifacts to the Bun save endpoint and added a regression test proving HyperEdit comment notes are included in the saved HTML payload.

  9. Accepted targeted live editing: users need to make small changes directly, but the frontend should record focused deltas or durable edit objects rather than treating the whole page as editable source.
  10. Replaced whole-page contenteditable activation with targeted block activation: edit mode turns on tools, and text input requires activating a specific block.
  11. Verified the current behavior with bun run check and the Playwright test suite.

Standing decisions

Accepted

No build step yet

Keep the runtime source directly loadable in the browser. Do not add a bundler, transpiler, framework migration, or generated output for this phase.

Accepted

Typed JS first

Use JSDoc and static checks before considering TypeScript files or a compile pipeline.

Accepted

Minimal modularity

Split code only when the boundary is already obvious and the split can use browser-native ES modules.

Prototype only

Full DOM snapshots

Full-document HTML save remains acceptable for the prototype, but it is not the long-term source of truth.

Accepted

Durable annotations

User comments are editor feedback stored with the HTML artifact, rendered into the page, and kept separate from source text so the agent can safely update the document.

Accepted

Targeted editing

Live document editing is required, but whole-page edit mode is the wrong boundary. User edits should become focused deltas the agent can reconcile with the source.
